## MultiAgent 工程范式

定义了main 三万，创建2个子agent  四万，五万。 

| 文件 | 主 Agent (main) | 子 Agent (siwan/wuwan) | 原因 |
|------|:---:|:---:|------|
| **AGENTS.md** | ✅ | ✅ | 项目规范所有 agent 都需遵守 |
| **SOUL.md** | ✅ | ✅ | 人格可以继承或覆盖 |
| **TOOLS.md** | ✅ | ✅ | 工具规范通用 |
| **IDENTITY.md** | ✅ | ✅ | 身份定义可以有自己的版本 |
| **USER.md** | ✅ | ✅ | 了解用户偏好对子 agent 也有用 |
| **HEARTBEAT.md** | ✅ | ❌ | 心跳轮询只有主 agent 才做，子 agent 是任务驱动的 |
| **BOOTSTRAP.md** | ✅ | ❌ | 新手引导只需主 agent，子 agent 无需 onboarding |
| **MEMORY.md** | ✅ | ❌ | 主 agent 的全局备忘录，子 agent 有自己的 workspace-<id>/MEMORY.md |


IDENTITY 是名片（"你是谁、擅长什么"），SOUL 是灵魂（"你怎么说话、怎么想"）。IDENTITY 影响的是角色认知，SOUL 影响的是表达风格。

main loadWorkspaceBootstrapFiles: AGENTS → SOUL → TOOLS → IDENTITY → USER → HEARTBEAT → BOOTSTRAP → MEMORY

子agent 只加载 MINIMAL_BOOTSTRAP_ALLOWLIST，5个。

### 多Agent并行执行


```
OpenClaw 的多 agent 并行基于 **Spawn → 独立执行 → Auto-Announce** 三步模型


参考unix posix_spawn() / fork() 概念：非阻塞地创建一个有独立生命周期的执行单元。

首先是 Plan和Decide 策略。 
Supervisor (三万)  callGateway 调用 sessions_spawn() 生成子 agent
       │
       │  不阻塞，立即返回 { childSessionKey `agent:${agentId}:subagent:${uuid}`, runId }， registerSubagentRun
       │
       ├──→ 子 agent 在独立 session 中异步执行 ReAct 循环
       │
       └──  子 agent 完成后，自动 announce 结果回 Supervisor

         │              │              │
         ▼              ▼              ▼
    ┌─────────┐   ┌─────────┐   ┌─────────┐
    │ 四万(1)  │   │ 四万(2)  │   │ 五万(3)  │    ← 3 个独立进程/session
    │ vLLM    │   │ TGI     │   │ SGLang  │       并行执行
    │         │   │         │   │         │
    │ ReAct:  │   │ ReAct:  │   │ ReAct:  │
    │ search→ │   │ search→ │   │ search→ │
    │ read→   │   │ read→   │   │ exec→   │
    │ exec→   │   │ exec→   │   │ read→   │
    │ 总结    │   │ 总结    │   │ 总结    │
    └────┬────┘   └────┬────┘   └────┬────┘
         │              │              │
         │ announce     │ announce     │ announce
         ▼              ▼              ▼
┌──────────────────────────────────────────────────────────────────────────┐
│ 三万 (Supervisor)                                                        │
│                                                                          │
│  收到 3 条 announce 消息 (作为 user message 注入)


                    ┌─ mode="run" ────────────────────────┐
                    │                                      │
Supervisor ──spawn─→│  子agent 执行 → announce → 归档/删除 │
                    │  (fire-and-forget)                   │
                    └──────────────────────────────────────┘

                    ┌─ mode="session" ────────────────────┐
                    │                                      │
Supervisor ──spawn─→│  子agent 绑定 thread → 持续可对话    │
                    │  用户可直接在 thread 中追问            │
                    │  session 保留，不自动清理              │
                    └──────────────────────────────────────┘
需要"驻场"的用 session，保留对话。

triggerMessage然后 通过Gateway给 三万。 
direct: callGateway({ method: "agent", sessionKey: 三万的key })
直接注入三万的 session，触发新一轮 run。
三万收到后，作为 user message 出现在对话中，三万可以继续 ReAct 处理或等待其他子 agent。 

数据隔离： 每个 agent 有独立的 workspace、session、memory、向量库。  唯一通道: announce 机制 (结果文本单向推送给 parent) 。

注意： 对于Supervisor， 每个 announce 到达时触发一次新的 run。
Think -> Act -> Observe -> Think -> Stop

Think-> 收到 1/3 的结果，还需等待 2 个 -> Stop
Think: 3/3 结果全部收齐，开始综合  -> 输出给用户  -> Stop


subagent-spawn.ts


```

## ReAct Loop

```

runEmbeddedPiAgent
    runEmbeddedAttempt run/attempt.ts
         activeSession.prompt(effectivePrompt) 启动核心Loop 
           标准的ReAct Loop.

处理 attempt 结果 → 返回给 Gateway → 触发 announce (如果是子 agent)

约束控制：
超时: scheduleAbortTimer
Context window: 消息太长时 SDK 触发 compaction (摘要压缩)，如果压缩后仍超出，退出
```
