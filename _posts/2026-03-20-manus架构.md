### ReAct Loop

```
Analyze → Plan → Execute(单个 action) → Observe → Append to Event Stream → 下一轮

和OpenClaw差异， Execution Engine — CodeAct 范式：
用可执行的 Python 代码作为统一 action 格式：JSON Tool Call 出错时，Agent 得到的是一个模糊的 error string，它必须"猜"问题在哪。而 CodeAct 的 Python traceback 精确告诉了错误类型、行号、变量状态——这些信息对 LLM 来说极易理解和修复。

单个 CodeAct 可以完成之前 5 轮 JSON 调用的工作量。这直接减少了 LLM 推理次数，意味着更低的延迟和成本。

```

### Context & Memory

文件系统作为"终极上下文"，和OpenClaw的 L2 L3 markdown as database 设计类似。 
```
Context Window 内                          Context Window 外
┌────────────────────────┐                ┌────────────────────────┐
│                        │                │                        │
│  System Prompt (稳定)   │   ──写入──▶    │  File System (∞)       │
│         ▼              │                │  context.md / notes.md │
│  Tool Definitions (固定)│   ◀──按需读──  │  task_plan.md          │
│         ▼              │                │  工具结果全文            │
│  Event Stream          │                └────────────────────────┘
│  [act₁][obs₁]...[actₙ] │
│  (append-only, 含错误)  │                ┌────────────────────────┐
│         ▼              │                │  Sub-agent VMs         │
│  Compact 引用          │   ──spawn──▶   │  (独立上下文，线性扩展)  │
│  (旧结果 → URL/路径)    │   ◀──结果──    │            │
│         ▼              │                └────────────────────────┘
│  Todo / Planner 输出    │
│  (目标锚定在末尾)       │
│                        │
└────────────────────────┘
本质 ：
    短期记忆： Event Stream (即时上下文)，Append-only 的 action/observation 时序日志，充分利用 prefix caching. 
    长任务内的文件记忆：按需 glob/grep 检索，  **和OpenSpec project.md spec.md  tasks.md 范式类似**
        File System as Context（文件系统即上下文）
  ┌─────────────────┐ ┌──────────────┐ ┌──────────────┐
  │  context.md      │ │  notes.md    │ │ task_plan.md │
  │  上下文感知状态   │ │  研究笔记     │ │  执行路线图   │
  │  用户偏好跟踪     │ │  中间结果存储 │ │  步骤分解     │
  └─────────────────┘ └──────────────┘ └──────────────┘

Agent 不断重写 todo.md → 全局计划被"朗诵"到上下文末尾
      → 自然语言偏置模型的注意力 → 保持目标对齐， 保存 多次steps后 目标不丢。
```

### Manus 对Agent 优化方式

```
生产环境 Agent 的 Token 比例：

Input Tokens  ████████████████████████████████████████  100x
Output Tokens █                                          1x

→ 输入上下文主导了成本和延迟
→ **KV-Cache 命中率是生产 Agent 的第一性能指标**



  很多和 推理 和 时延 相关的优化： 
  1. CodeAct 减少 steps，效率的优化
  2. **Context Engineering**

    
    很重要的一块在这里。 优化Agent的长事务，需要深度考虑 推理引擎的特性 来设计Context.
    - subAgent Context Isolation 上下文隔离，专一精准和长度控制。
      相对小的promt, prefill时间会短. vllm会有 chunked prefill影响。
    -   Append-only 的 action/observation 时序日志， 利用 prefix kv caching.
       我们知道 对于vllm/sglang 推理来说，prefix caching特别有用，Manus很多优化基于这个。 

       Stable Prefixes（稳定前缀）。
       Append-Only Context（仅追加上下文）

       ✅ 正确做法：只追加，永不修改
[action_1] [obs_1] [action_2] [obs_2] [action_3] [obs_3]
                                       ↑ 新增部分，前缀缓存保持有效
        正确做法：保持工具列表不变，用 masking 控制可用性
    - 利用 LLM 对近期 token 的注意力偏差（对抗 lost-in-middle）
       prompt 分层中，考虑到llm的头 尾 高attention 特性。

    - MCP 工具通过 CLI 暴露，Agent 用 Bash 执行。
     和skills类似，避免mcp大量上下文toolspec浪费。 

    - 动态压缩
     Summarization（摘要化） Compaction（紧凑化）

     所有压缩策略都是**可恢复的（restorable）**，可以重新ref  load. 

本质，在实际中 ** 知行合一优化。 ** 
     https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus
```

```
Cloudflare:
  Worker (V8 Isolate):  ~0ms (已部署) / ~5ms (冷启动)   没有完整cli环境

E2B (Cloud):
  按 sandbox 存活秒数计费
  ~$0.10 / sandbox-hour (基础)
  适合: 短生命周期、按需创建销毁
```

## Manus 和 OpenClaw架构特性对比
```
云端 SaaS 和 自托管Pc. 
多 Agent 编排层 + E2B 沙箱, 单 Gateway 守护进程 + 嵌入式 Agent Runtime. 

Manus是云端的 "虚拟电脑" — 完整 OS 环境执行任务，并发沙盒，处理复杂任务能力强很多，scale规模可以很大。 

记忆， Manus可以基于 完整的file system， OpenClaw 在jsonl和 md上会引入 vectorstore补充。 

主要差异是1个基于task， 一个是个人助理 要养，长期记忆部分需要搜索。
```
