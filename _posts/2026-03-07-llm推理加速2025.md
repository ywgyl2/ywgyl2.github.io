# 压缩token输入

LLMLingua-2原理：
通过 GPT-4 的数据蒸馏训练 一个小型编码器（BERT 级或轻量 LLM encoder），用来打分和预测，从而压缩 context。 

推理阶段（在线压缩）
- 将用户指令 + 系统提示 + 历史对话 + RAG 检索出来的长文档等，拼成一整段长 prompt，分词得到 `tokens`。
- 小编码器一次前向，得到每个 token 的重要性得分或 `keep / drop / compress` 预测。
- 结合一个压缩预算（比如「最多只保留 30% token」），用贪心 / 动态规划等方法，选出要保留的关键 token，对评分低的 token 做删除或段落级摘要。
- 得到压缩后的 `tokens'`，再送入大模型推理，注意这里大模型本身的权重完全不改，只是「吃到的上下文」变短了。

```python
def llmlingua2_compress(prompt: str, budget_ratio: float):
    # 1. 分词
    tokens = tokenizer(prompt)

    # 2. 小编码器打分（一次前向，开销远低于大模型）
    logits = encoder(tokens)                 # [seq_len, num_classes]
    scores = importance_from_logits(logits)  # [seq_len]

    # 3. 在给定 budget 下选出要保留的 token
    budget = int(len(tokens) * budget_ratio)
    selected_indices = select_topk_under_budget(scores, budget)

    # 4. 根据选择重建压缩后的 prompt（可选：对连续被丢弃的片段做摘要）
    compressed_tokens = [tokens[i] for i in selected_indices]
    return detokenize(compressed_tokens)

# 实际调用链路示意
long_prompt = build_long_prompt(user_query, history, retrieved_docs)
compressed_prompt = llmlingua2_compress(long_prompt, budget_ratio=0.3)
answer = big_llm_generate(compressed_prompt)
```

多模态 文中 也有很多压缩方法。 

# 量化

本质是 推理时完全是普通的矩阵乘 + 标量缩放。 直接用整型 GEMM**（如 INT4/INT8 matmul），在输出端乘以 scale 做一次轻量级反量化。 

vLLM 的 FP8 Per‑Tensor，默认一组scale， 分布差异，会丢失精度， 质量会差。 
Per‑Channel / Per‑Block / Group：以「行列 / 通道 / 小块」为单位，各自统计分布并分配独立的 scale，能更细粒度适配权重的动态范围，是当前 LLM 权重量化中最主流、质量–性能折中最好的一类做法。 
 Per-Channel （按行/列量化，每行一个 scale）
Per‑Block（按小块量化）把大矩阵沿着列或行切成小块，比如「每 128 列为一个 block」

Per‑Group（按列 group 量化，GPTQ/AWQ 的核心粒度）
GPTQ/AWQ 属于PTQ（Post‑Training Quantization），常用。 

为了保持精度， 往往**权重量化 + 激活/KV 保持 FP16/FP8**。 

```
      Per-Block 示意图（FP8 block_size=128）

      W = [4096 × 4096] 的权重矩阵

               col 0-127      col 128-255     col 256-383    ...   col 3968-4095
             ┌─────────────┬──────────────┬──────────────┬───┬──────────────┐
      行 0   │  block(0,0) │  block(0,1)  │  block(0,2)  │...│  block(0,31) │
      ~      │  scale_0_0  │  scale_0_1   │  scale_0_2   │   │  scale_0_31  │
      行 127 │  128×128个  │  128×128个   │  128×128个   │   │  128×128个   │
             ├─────────────┼──────────────┼──────────────┤   ├──────────────┤
      行 128 │  block(1,0) │  block(1,1)  │  block(1,2)  │...│  block(1,31) │
      ~      │  scale_1_0  │  scale_1_1   │  scale_1_2   │   │  scale_1_31  │
      行 255 │             │              │              │   │              │
             ├─────────────┼──────────────┼──────────────┤   ├──────────────┤
             │    ...      │     ...      │     ...      │...│     ...      │
             ├─────────────┼──────────────┼──────────────┤   ├──────────────┤
      行3968 │ block(31,0) │ block(31,1)  │ block(31,2)  │...│ block(31,31) │
      ~      │ scale_31_0  │ scale_31_1   │ scale_31_2   │   │ scale_31_31  │
      行4095 │             │              │              │   │              │
             └─────────────┴──────────────┴──────────────┴───┴──────────────┘

      总 block 数: (4096/128) × (4096/128) = 32 × 32 = 1024 个 block
      总 scale 数: 1024 个（每个 FP32 = 4 bytes → 额外 4KB，vs 权重本身 16MB）

      对比:
        Per-Tensor  →   1 个 scale  → 精度差
        Per-Block   → 1024 个 scale → 精度好，额外开销 0.025%
  
```

```
      Per-Group 示意图（group_size=32，INT4 量化）

      W 的第 0 行 (out_feature=0)，共 4096 个权重:

      ┌── group 0 ──┐┌── group 1 ──┐┌── group 2 ──┐         ┌── group 127 ─┐
      │ w0 w1 .. w31 ││w32 w33..w63 ││w64 w65..w95 │  ...    │w4064 .. w4095│
      │  scale_0     ││  scale_1    ││  scale_2    │         │  scale_127   │
      │  zero_0      ││  zero_1     ││  zero_2     │         │  zero_127    │
      └──────────────┘└─────────────┘└─────────────┘         └──────────────┘

      每行 128 个 group × 4096 行 = 524,288 组 scale+zero
      额外存储: 524,288 × (2+2) bytes ≈ 2MB（vs 原始权重 32MB，开销 ~6%）

      具体例子（group 0 的 32 个权重）:

      原始 FP16:  [0.12, -0.45, 0.33, 0.67, ..., -0.89]  (32个数)
      统计:       min = -0.89, max = 0.67
      INT4 范围:  [0, 15]（无符号）或 [-8, 7]（有符号）

      非对称量化 (unsigned INT4):
        scale_0 = (0.67 - (-0.89)) / 15 = 0.104
        zero_0  = round(-(-0.89) / 0.104) = round(8.56) = 9

        量化:     w_q[i] = clamp(round(w[i] / scale_0) + zero_0, 0, 15)
        反量化:   w[i]  ≈ (w_q[i] - zero_0) × scale_0

        0.12  → round(0.12/0.104) + 9 = 10  → (10-9)×0.104 = 0.104  (误差 0.016)
       -0.45  → round(-0.45/0.104)+ 9 = 5   → (5-9)×0.104 = -0.416  (误差 0.034)
        0.67  → round(0.67/0.104) + 9 = 15  → (15-9)×0.104 = 0.624  (误差 0.046)
```

# Sparse Attention（块稀疏注意力）

把 KV 序列按时间或位置切成若干个 block（如每 16/32/64 个 token 一块），对每个查询 token，不再对「所有 block」做注意力，而是只对**少数几个高相关 block**做注意力（其余视为 0），从而把注意力矩阵从稠密变成**块级稀疏**，配合**周期性的密集修正**。

类比：把图书馆按「书架」分区（每个书架 = 一个 block），用一张**索引卡片**快速扫一眼每个书架的主题摘要，只走进**最相关的 2–3 个书架**仔细翻，其余书架直接跳过。
每隔一段时间，回头把整个图书馆快速巡检一遍，防止之前跳过的书架里藏着你后来需要的关键信息。

```
密集 vs 块稀疏注意力矩阵（8 个 block 的例子）

标准密集注意力（每个 query 看所有 block）：
        KV block →   B0   B1   B2   B3   B4   B5   B6   B7
  Query token q  │  [✓]  [✓]  [✓]  [✓]  [✓]  [✓]  [✓]  [✓]   ← 8 个块全算
                    ──────────────────────────────────────────
                    内存读 = 8 块 × block_size 个 KV

块稀疏注意力（只选 Top-2 + 最新 1 块 = 3 块）：
        KV block →   B0   B1   B2   B3   B4   B5   B6   B7
  Query token q  │  [ ]  [✓]  [ ]  [ ]  [✓]  [ ]  [ ]  [✓]   ← 只算 3 块
                    ──────────────────────────────────────────
                    内存读 = 3 块 × block_size 个 KV  → 省了 62.5%

  选中依据:
    B1  → 历史相关性最高（比如包含关键实体）
    B4  → 次高相关
    B7  → 最新上下文（永远保留）


```

```
假设 prompt 已预填充，KV cache 里已有 16 个 token，分成 4 个 block：
  B0=[t0 t1 t2 t3]  B1=[t4 t5 t6 t7]  B2=[t8 t9 t10 t11]  B3=[t12 t13 t14 t15]

─── Step 1: 生成 t16 ────────────────────────────────────────
  1) 对 4 个 block 算「块级摘要分数」:
       B0=0.1  B1=0.8  B2=0.3  B3=0.9（B3 最新+最相关, B1 也很相关）
  2) Top-2 选中: B1, B3
  3) 只在 B1 和 B3 的 8 个 KV 上做注意力（而不是全部 16 个）
  4) 输出 → t16,  更新 KV cache: B4=[t16]
  ➜ 计算量: 8/16 = 50%

─── Step 2: 生成 t17 ────────────────────────────────────────
  块分数可能变了（因为 query 不同了）:
       B0=0.05  B1=0.6  B2=0.7  B3=0.85  B4(新)=0.9
  Top-2 选中: B3, B4（最新两块最相关）
  只在 B3 + B4 上做注意力
  输出 → t17,  B4=[t16, t17]
  ➜ 计算量: 更少了（B4 才 2 个 token）

─── Step 3: 生成 t18 ────────────────────────────────────────
  Top-2: B3, B4      → 稀疏注意力
  输出 → t18,  B4=[t16, t17, t18]

─── Step 4: 生成 t19 ────────────────────────────────────────
  Top-2: B1, B4      → 稀疏注意力（B1 又变得相关了）
  输出 → t19,  B4=[t16, t17, t18, t19]（满了）

  ★ 触发密集修正（第 4 步，f=4）★
  用密集注意力对 B0~B4 全部 20 个 KV 重算一遍，刷新 KV cache
  → 纠正前 4 步稀疏近似积累的偏差

─── Step 5~8: 又开始稀疏解码 → Step 8 再次密集修正 ──────
  ... 循环往复 ...
```

```python
def decode_with_rectified_sparse_attn(model, prompt_tokens, steps, b=32, f=16, top_p=0.25):
    # 1. 预填充：标准密集注意力，初始化 KV cache
    kv_cache = dense_prefill(model, prompt_tokens)

    generated = []
    for t in range(steps):
        # 2. 稀疏阶段：基于当前 GQA 组的统计，选出需要关注的 block
        blocks = partition_kv_into_blocks(kv_cache, block_size=b)     # [num_blocks, ...]
        scores = estimate_block_scores(blocks)                         # 估算每块重要性
        selected = select_top_p_blocks(scores, top_p=top_p)           # 只保留 top‑p 的块

        # 3. 在选中的块上做注意力（组块稀疏注意力）
        next_token, kv_cache = sparse_step(model, kv_cache, selected_blocks=selected)
        generated.append(next_token)

        # 4. 周期性密集修正
        if (t + 1) % f == 0:
            kv_cache = dense_rectification(model, kv_cache)

    return generated
```

# Dynamo
调度 + 内存管理中间层，LB和 vllm/sglang之间。 
 Planner（自动扩缩容），转发到最优 worker。 
 KVBM（KV Block Memory Manager）， 统一管理所有 worker 的 KV cache， 向 Router 上报「哪些 prefix hash 在我这有缓存」。 

 **本质 基于上报，在router上能够 更好路由 利用 prefix caching。** 

 **请求路由感知**：Dynamo 监听每个推理后端的 KV cache 信息（存了哪些 prefix hash），接收到新 request 时，对 prompt 算 hash 并和各 worker 缓存匹配。

# PD分离

PD 分离主要解决 **prefill 干扰 decode 耗时**的问题。
- Prefill 是**计算密集型**：一个长 prompt（如 4096 token）的 prefill 需要一次性把所有 token 过完所有层，GPU 算力被占满；
- Decode 是**访存密集型**：每次只算一个 token，但需要读大量 KV cache；
- 当一个长 prefill 正在跑的时候，同一个 GPU 上等待 decode 的请求只能排队，导致 TPOT（Time Per Output Token）抖动变大。


```
═══ PD 分离（P 和 D 在不同 GPU 上）═══

P-GPU 时间线（只跑 Prefill）:
  ┌─────────────────────┬─────────────────────┬────────...
  │   Prefill 请求A     │   Prefill 请求B     │
  │   (4096 tokens)     │   (2048 tokens)     │
  └─────────────────────┴─────────────────────┴────────...
  t=0              t=80ms                t=120ms

  Prefill 完成后，KV cache 传输到 D-GPU  ───→
                                              ↓
D-GPU 时间线（只跑 Decode）:                  收到 KV
  ┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┬──...
  │D │D │D │D │D │D │D │D │D │D │D │
  └──┴──┴──┴──┴──┴──┴──┴──┴──┴──┴──...
  每个 decode step ≈ 10ms, 不被任何 prefill 打断
  → TPOT 稳定在 10ms 左右
```

```


═══ 形态 1: vLLM 默认混合部署（Continuous Batching）═══

  一个 vLLM 实例, 一个 GPU, 一个 Scheduler
  ┌─────────────────────────────────────────────────────┐
  │ Scheduler 每个 iteration 组一个 batch:               │
  │                                                     │
  │  iteration 1 的 batch:                              │
  │  ┌─────────────────────────┬────┬────┬────┐        │
  │  │ 请求A prefill (新来的)    │B.d │C.d │D.d │        │
  │  │ 2048 tokens             │1tok│1tok│1tok│        │
  │  └─────────────────────────┴────┴────┴────┘        │
  │  所有请求的 prefill 和 decode 在同一次 forward 里跑    │
  │                                                     │
  │  问题: A 的 2048 token prefill 让这个 iteration 变慢  │
  │  B/C/D 只需算 1 token 但被迫等 A 的大矩阵算完         │
  │  → decode 延迟被 prefill 「拖慢」                     │
  └─────────────────────────────────────────────────────┘
1. GEMM 的总耗时 ∝ 行数. 2051 行的 GEMM 比 3 行的 GEMM 慢得多
     如果没有 A 的 prefill, B/C/D 这 3 行的 GEMM 只要 ~0.1ms
     加上 A 变成 2051 行, 整个 GEMM 要 ~5ms, B/C/D 也要等 5ms
  2. Attention 阶段更严重: A 要算 2048×2048 的 attention 矩阵,
     GPU SM 被大量占用, B/C/D 的 1×N attention 被排在后面
  3. 显存带宽竞争: A 的 prefill 疯狂读写中间激活,
     B/C/D 读 KV cache 的带宽被挤压

"拖慢" 的本质是: **大 GEMM 的耗时由最大的那个请求决定,小请求虽然同时在算, 但要等整个大 GEMM 结束才能拿到结果**。

**Chunked Prefill（vLLM 的实际解法）本质就是 切碎**。


═══ 形态 2: 完全 PD 分离（独立实例，独立 GPU）═══

  P-GPU 0 (独立进程)              D-GPU 1 (独立进程)
  ┌──────────────────┐           ┌──────────────────┐
  │ 模型权重 (14GB)   │           │ 模型权重 (14GB)   │
  │ 只跑 Prefill      │ ──KV──→  │ 只跑 Decode       │
  │ 算完就丢给 D      │  网络传输  │ 从不跑 Prefill    │
  └──────────────────┘           └──────────────────┘

  优点: Decode 永远不被 Prefill 打断, TPOT 非常稳
  缺点: 权重存两份; KV 要跨网络传; P/D 各自有空闲时段
```
