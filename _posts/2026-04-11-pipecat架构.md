## Background
```
快速了解下 pipecat的设计。 

Pipecat 本质上是 1个实时语音/多模态 pipeline 编排层，本质处理 pipeline orchestration，实时编排：

- 音频输入
- ASR/STT 转写
- 用户 turn 检测
- 上下文聚合
- 调用外部 agent / LLM
- 把文本结果送给 TTS
- 回传到传输层
```

## 框架执行模型
```
一个会话构建 一个 Pipeline + PipelineTask，隔离工程更简单可控。  
    **帧没有会话标识**：Frame 不带 session_id，管道内部没有路由能力。
```
```text
transport.input
  -> STT
  -> LLMUserAggregator
  -> LLM
  -> TTS
  -> transport.output
  -> LLMAssistantAggregator
```
```
**这个pipline是个职责链，工厂方法实时按需各种实现**。 

围绕 Frame 流水线 + 每个处理器自己的异步队列/状态机 构建。 
- 围绕 `FrameProcessor` 传递 `Frame`
- `Pipeline` 把多个 processor 串起来
- `PipelineTask` 负责跑整条链、加 observer、心跳、idle 监控、生命周期管理

Pipeline 只是线性连接器， **线性保持 结构简单，工程简单 可排查**。 
- 加一个 source
- 加一个 sink
- 把中间 processors 线性 link 起来

`PipelineTask` 主要是生命周期管理：

- 初始化 task manager / clock / observer
- 包装最终 pipeline
- 监听 start/end/idle
- 管理 heartbeat、turn tracking


2层queue：
1. `__input_queue = FrameProcessorQueue()`
2. `__process_queue = asyncio.Queue()`

所有 frame 先进入 `__input_queue`， 先处理 system frame，普通 frame 进入 `__process_queue` 排队执行。 

结果： 
- 普通音频帧、文本帧按常规顺序处理
- `InterruptionFrame`、`CancelFrame`、`StartFrame` 这种 `SystemFrame` 会优先出队
```

## 数据流和控制流
```text
1. 用户说话
   InputAudioRawFrame 连续进入 pipeline

2. VAD / 转写到来
        
   VADController 产生 VADUserStartedSpeakingFrame
   STT 产生 InterimTranscriptionFrame（实时中间结果，用于前台显示） / TranscriptionFrame （最终确认结果）

3. User turn 被打开
   UserTurnController 判定 user turn started
   LLMUserAggregator 开始聚合这一轮用户文本

4. 用户说完
   VADUserStoppedSpeakingFrame 到来
   TurnAnalyzer + STT 最终转写共同判定 turn stop
    本质 默认不是“来一段转写就立刻发给 LLM”，需要累积一个 user turn 的文本。 

5. 用户消息入上下文
   LLMUserAggregator push_aggregation()
   context.add_message({role: user, content: ...})
   然后 push context frame 给 LLM

   送给 LLM 的默认单位不是“单个 transcription 片段”，而是“一个完整 user turn”

6. LLM 生成
   LLMFullResponseStartFrame (ControlFrame)
   LLMTextFrame...   × N  (DataFrame)  每个 token/chunk 一帧，流式产生  
   LLMFullResponseEndFrame(ControlFrame) 标记一次 LLM 响应的结束边界
   
   本质： LLM loop中 中间文字会被 TTS 说出。 

7. TTS 发声
   TTS 把文本切分/聚合成 sentence/token  
   创建 audio context
   输出 TTSStartedFrame  (ControlFrame)   / TTSAudioRawFrame  × N  (DataFrame, 继承 OutputAudioRawFrame) 实际的 PCM 音频数据块，连续流式产生     / TTSTextFrame  (DataFrame, 继承 AggregatedTextFrame， 用它把"真正被说出来的文字"写回 context)   / TTSStoppedFrame (ControlFrame) 

8. assistant 消息入上下文
   LLMAssistantAggregator 在 assistant turn 完成后把真正输出的 assistant 内容写回 context。 
   LLMContextSummarizer 压缩管理 上下文。 

9. 空闲阶段
   BotStoppedSpeakingFrame 后，如果用户没说话，也没 function call
   UserIdleController 才开始 idle 计时
```
```
各种帧 通过优先级流动： 
1. 数据帧：比如 `InputAudioRawFrame`、`TranscriptionFrame`、`LLMTextFrame`、`TTSAudioRawFrame`。
控制帧：
2. turn 帧：比如 `VADUserStartedSpeakingFrame`、`UserStartedSpeakingFrame`、`UserStoppedSpeakingFrame`。
3. 生命周期帧：比如 `StartFrame`、`EndFrame`、`CancelFrame`。
4. 打断和调度帧：比如 `InterruptionFrame`、`HeartbeatFrame`、`SpeechControlParamsFrame`。

普通数据流默认向 downstream 走，但某些状态信号，比如 `BotStartedSpeakingFrame`、`InterruptionFrame`、`FunctionCallResultFrame`，可以向 upstream 或双向广播。
```
## 连贯语意
```
1. `turn 边界`
    “这一轮用户表达什么时候开始、什么时候结束”

- VAD 检测到停止 （判断“有没有在说话”）
- turn analyzer 结合音频内容判断“是不是完整说完”
- finalized transcript 或 STT timeout 做最终收口
    VADUserStartedSpeakingFrame：告诉 analyzer 用户开始说了
    VADUserStoppedSpeakingFrame：告诉 analyzer 用户停了
    TranscriptionFrame：看转写是否已经出来、是否 finalized
    STTMetadataFrame：拿 STT 的 p99 延迟，决定要不要再等一下最终转写

VAD 停止，调用 analyze_end_of_turn()，看 analyzer 返回的 EndOfTurnState 是不是 COMPLETE。
analyzer 认为“可能说完了”，还要再等：
      - finalized transcript 到来，或者
      - 一个基于 STT 延迟的 timeout
条件都满足后，才真正触发 user_turn_stopped。


  VAD 停了
    -> turn analyzer 说“像是说完了”
    -> ASR 再补最后的 finalized transcript
    -> 然后才真正 stop turn
核心策略在TurnAnalyzerUserTurnStopStrategy。 

2. `transcription 聚合`
_aggregation buffer聚合。

3. `最终转写/延迟保护`
VAD检查后
`TurnAnalyzerUserTurnStopStrategy` 会等：

- turn analyzer 确认 `turn_complete`
- finalized transcript
- 或基于 `STTMetadataFrame.ttfs_p99_latency` 的 timeout
  
  避免：

- 用户只是短暂停顿，就被过早截断
- STT 最终词还没完全出来，就把半句发给 LLM

总结： 
- 先形成稳定 user turn
- 再把这轮内容交给 LLM

- VAD 的快
- ASR 的准
- turn analyzer 的稳
```
## 打断监测 控制流 到 数据流的协同
```
本质 通过优先队列实现中断的零延迟响应。

SystemFrame优先级高于普通 frame。
打断本质是 广播一个高优先级 `InterruptionFrame`。

self.broadcast_interruption()  **本质沿着管道 传播性 链路清理**。  
   __reset_process_queue()    ← 清空自己的 __process_queue queue 清空。 
   self.broadcast_frame(InterruptionFrame)  ← 向上游+下游各推一个 InterruptionFrame
   
   downstream InterruptionFrame ──→ 沿管道链依次到达每个下游 Processor
      每个 Processor 收到后调用 _start_interruption()：
        如果当前正在处理 UninterruptibleFrame → 只清空队列不杀 task
        否则 → cancel process_task + 清空 queue + 重建 process_task
      效果：TTS 停止合成、OutputTransport 停止播放、所有排队帧被丢弃

谁广播： 检测到打断事件的那个 Processor（通常是 STT/VAD/InputTransport）。
结果： 
整条管道上所有 Processor 的 process_queue 同时被清空（保留 UninterruptibleFrame），正在处理的帧被 cancel，管道快速恢复空闲状态以接收用户新输入。


`VAD/UserTurnController -> InterruptionFrame -> TTS/assistant aggregator/transcript/idle controller` 多模块联动。

`LLMUserAggregator` 内部同时挂了：

- `UserTurnController`
- `UserIdleController`
- 可选 `VADController`
```
```text
Bot 正在说话
  -> LLMTextFrame ...
  -> TTS 正在输出 TTSAudioRawFrame ...

用户突然开口
  -> InputAudioRawFrame
  -> VADController 识别 speech started
  -> VADUserStartedSpeakingFrame
  -> UserTurnController 判定 user turn started
  -> LLMUserAggregator._on_user_turn_started
  -> broadcast_interruption()
  -> InterruptionFrame 向上下游广播

InterruptionFrame 到各模块后： （**数据流的协同**）
  -> TTS: 终止 audio context，停止继续发声
  -> AssistantAggregator: 收口当前 assistant turn，reset
  -> TranscriptProcessor: 发出“被打断前已说出的部分”
  -> UserIdleController: 因用户 turn in progress，不启动 idle timer

然后：
  -> 新一轮用户音频/转写继续进入
  -> 用户 turn 正常聚合
  -> 结束后再触发新的 LLM run
```

```
总结： 
**打断既要停声音，也要停“未完成的 assistant 聚合”，但不能把已经完成的上下文弄脏**
TTS 充当“最终是否写回 assistant context 的仲裁层”。 依赖TTSTextFrame。 
```

## 管道通畅性追踪

```
1. HeartbeatFrame 作为 ControlFrame 穿越整条管道，Monitor 检测穿越耗时和超时。
每 1s 注入心跳，超过 10s 未到达 Sink 则 WARNING
2. `IdleFrameObserver` 监听管道中所有 Processor push 的帧，如果指定的帧类型（默认 BotSpeaking + UserSpeaking）在超时时间内都没出现，触发 `on_idle_timeout` 事件并可选自动 cancel。
3. Metrics + Observer（性能度量）
   
| 指标 | 含义 | 来源 |
|------|------|------|
| `TTFBMetricsData` | Time-To-First-Byte（首字节延迟） | STT/LLM/TTS |
| `ProcessingMetricsData` | 单次处理耗时 | 各 Processor |
| `LLMUsageMetricsData` | Token 用量 (input/output) | LLM |
| `TTSUsageMetricsData` | TTS 字符用量 | TTS |
```
**UserBotLatencyObserver** 专门测量用户 → Bot 响应延迟：

```
VADUserStoppedSpeaking → ... → BotStartedSpeaking
       ↑ 记录时间                      ↑ 计算差值 = 端到端延迟
```
OpenTelemetry 追踪： `TurnTraceObserver` 为每个对话轮次创建 OpenTelemetry Span
```
Conversation Span
 └── Turn 1 Span
      ├── STT processing (TTFB: 120ms)
      ├── LLM processing (TTFB: 450ms, tokens: 150)
      ├── TTS processing (TTFB: 80ms)
      └── User-Bot Latency: 650ms
 └── Turn 2 Span
      └── ...
```

## LLM 慢时的处理

```
httpx.TimeoutException 会触发 on_completion_timeout，最终也会触发LLMFullResponseEndFrame。 整条 pipeline 状态不会卡死。 
框架给稳定的状态边界和 frame/event 钩子。 Hook做业务逻辑。 
```

## 静默/空闲时的处理
```
Pipeline idle：整条流水线长时间没有效 frame。 
  IdleFrameObserver观察特定 frame 是否还在出现：
     - `BotSpeakingFrame`
      - `UserSpeakingFrame`
  UserIdleController 解决“bot 已经把球踢给用户，但用户不说话”。

  TTS 后推一小段静音 在 `TTSStoppedFrame` 后会补一段全零 PCM 静音帧。 
```
